# 平台化设计

- 状态：当前开发分支已实现第一阶段
- 日期：2026-07-17
- 面向：产品使用者、维护者与后续开发者

## 1. 目标与边界

构界从“能编辑条目的本地 Wiki”升级为“管理世界观生产全过程的创作平台”。平台化不等于引入账号、云端数据库或社区：当前核心用户仍是个人创作者，Markdown/YAML 仍是内容真源，SQLite 仍是可删除重建的索引。

平台层解决四类问题：创作者不知道全库当前状态；大量内容难以统一治理；反复创建相似条目成本高；条目之间的结构缺少全局视角。

本阶段明确不加入多人实时协作、远程托管、订阅计费和未经确认的 AI 自动写入。这些能力会改变隐私、部署与冲突模型，必须单独决策。

## 2. 产品结构

| 工作区 | 核心问题 | 当前能力 |
| --- | --- | --- |
| 创作总览 | 整个项目进展如何 | 规模、正史比例、草稿队列、连接数、内容成熟度、最近活动和关注队列 |
| 内容库 | 如何管理数百个条目 | 关键词、世界、类型和状态筛选；批量调整状态与标签 |
| 模板中心 | 如何复用创作方法 | 六套内置框架；世界库级自定义 YAML 模板；一键套用 |
| 时间线 | 事件如何排序 | 模糊时间范围、世界内显示文本与内部序号排序 |
| 关系网络 | 知识如何连接 | 全库节点、语义关系、正文提及、结构化关系台账 |
| 质量中心 | 哪些内容存在风险 | 断链、孤立条目、别名冲突、废弃引用和时空冲突 |
| 迁移与发布 | 如何跨设备和交付 | `.worldvault`、安全导入预览、冲突处理和静态审阅包 |
| 平台设置 | 如何管理本地工作区 | 多世界、重建索引、近期世界库、关闭、删除和退出 |

## 3. 核心工作流

### 3.1 从构想到正史

1. 在模板中心选择人物、地点、组织、事件、文化或规则框架。
2. 新条目默认进入草稿队列，补充正文、标签、时间和语义关系。
3. 在内容库筛选本轮条目，批量追加阶段标签或调整状态。
4. 在关系网络和质量中心检查缺失连接、断链与矛盾。
5. 确认后将条目转为正史，并通过审阅包进行只读检查。

### 3.2 世界库治理

创作总览中的“内容成熟度”是可解释的启发式指标：正文不少于 80 个字符，且至少具有标签或别名时，计为结构较完整。它用于发现待完善内容，不评价创作质量，也不会自动改变状态。

批量操作单次最多处理 200 个条目。服务会先解析所有目标，保存原始文件快照，再逐条原子写入并重建索引；任一步失败时恢复本批次文件，避免部分成功造成状态不一致。

### 3.3 模板沉淀

内置模板随应用发布，只读且不进入用户世界库。自定义模板写入 `templates/<template-id>.yaml`，属于用户内容的一部分，会进入完整 `.worldvault`。模板只为新条目提供初值，创建后条目不再依赖模板，因此删除模板不会破坏已有内容。

## 4. 技术接口

新增的平台 API 保持 Web 层薄、领域服务统一写入：

| API | 用途 |
| --- | --- |
| `GET /api/dashboard` | 返回聚合指标、分布、标签、最近活动和关注队列 |
| `POST /api/entries/bulk` | 批量调整状态、追加或移除标签 |
| `GET /api/templates` | 列出内置和世界库自定义模板 |
| `POST /api/templates` | 创建世界库自定义模板 |
| `DELETE /api/templates/{id}` | 删除自定义模板，拒绝删除内置模板 |
| `GET /api/graph` | 返回全库关系网络节点和边 |

仪表盘和关系网络数据均从可重建 SQLite 索引读取。自定义模板和条目正文从世界库读取，仍然可以在普通文本编辑器中维护。

## 5. 安全与失败处理

- 批量写入遵守现有乐观锁、原子文件替换和索引重建边界。
- 模板 ID、条目类型和状态在写入前校验，模板路径不能逃出世界库。
- 关系画布只渲染已转义的标题，Markdown 渲染仍过滤危险 HTML 和 URL。
- 全库关系 API 默认最多返回 250 个最近条目，硬上限 500，避免大型库直接拖垮浏览器。
- 任何索引损坏都可以通过“质量中心”或“平台设置”重建，不影响 Markdown 真源。

## 6. 后续演进

下一阶段优先级按实际使用阻力排序：

本阶段路线图已经完成：版本历史与恢复、原图和坐标分离的地图工作区、自定义字段 schema 与模板迁移、分支时间线比较与可解释合并，以及发送范围可见且只写提案区的可选 AI 建议层。

下一阶段不再预设为本次“全部功能”的隐含范围。多人协作、网络发布或新的模型适配应先形成独立需求与威胁模型。

多人协作只有在单人平台工作流稳定后再评估。若引入网络访问，必须同时设计身份验证、权限、CSRF 防护、审计日志和冲突解决，不能直接暴露当前本地服务。
