# 分发与数据迁移

- 状态：设计基线
- 日期：2026-07-16
- 受众：个人最终用户、构建与发布维护者

## 1. 目标与边界

用户下载程序、解压并运行后，应能立即创建、保存、浏览和审阅世界观，不需要预装 Python、Node.js、数据库服务或开发工具。发行包只包含程序、通用资源与明确标记的虚构示例标准副本，绝不包含作者或用户的真实世界观数据。

程序升级与内容迁移是两条独立路径：程序通过新的原生发行包升级；内容通过世界库目录或 `.worldvault` 传输包迁移。卸载或替换程序不能删除世界库。

## 2. 发行矩阵

| 产物 | 主要用户 | 首发状态 | 离线能力 | 验证要求 |
| --- | --- | --- | --- | --- |
| Windows x64 目录式 ZIP | 普通个人用户 | M1 必需 | 创建、编辑、检索、导入导出全部离线 | 干净 Windows 环境启动与真实工作流测试 |
| Linux x64 `tar.gz` | Linux 个人用户 | M3 | 与 Windows 基础能力相同 | 目标最低兼容系统原生构建和测试 |
| macOS `.app.zip` | macOS 个人用户 | 后续 | 与 Windows 基础能力相同 | Apple Silicon/Intel 分别评估，完成签名和原生测试 |
| wheel/sdist | Python 开发者 | 开发辅助 | 取决于可选功能 | 新虚拟环境安装并测试 CLI 与真实接口 |
| 容器 | 高级自托管用户 | 可选 | 基础功能离线 | 非 root 启动、健康检查和可写目录测试 |

普通用户首发采用“目录式压缩包”而不是单文件冻结程序。解压后双击启动仍属于下载即用，同时避免每次运行解压静态资源、受限临时目录无法执行等问题。

每个平台必须在对应操作系统构建。发布页明确 CPU 架构、最低系统版本、是否签名及能力差异，不使用模糊的“全平台通用”描述。

## 3. 发行包结构

```text
Worldbuilding-Wiki-0.4.0-windows-x64/
├── WorldbuildingWiki.exe
├── app/                    # 解释器、依赖、前端和内置模板
├── README.txt
├── LICENSE.txt
├── CHANGELOG.md
└── THIRD_PARTY_NOTICES.txt
```

发行包不得包含：

- `vault/`、`.worldvault` 或真实创作正文；
- `.env`、API 密钥、认证信息；
- `runtime/`、SQLite 索引、日志和崩溃转储；
- `.git/`、源测试缓存或构建机绝对路径；
- 仅在开发机存在、首次运行才下载的必要资源。

内置空白模板与“潮汐档案 · 示例全套数据”随正式发行包提供。示例标准副本属于只读应用资源；只有用户在新建页保持勾选或点击“载入示例”后，才会复制到所选世界库。删除示例只匹配固定示例标记，不影响用户数据。

## 4. 首次运行

启动器执行以下流程：

1. 检查应用资源完整性和可写用户目录。
2. 创建配置与运行状态目录，但不在用户选择前自动创建世界库。
3. 显示“新建世界库”“打开现有目录”“导入 `.worldvault`”三个入口。
4. 用户选定世界库位置后保存位置引用；新建时默认复制示例标准副本，也可取消勾选创建空库。
5. 启动回环 Web 服务并打开浏览器；浏览器关闭不等于服务停止，应用提供明确退出入口。

若默认端口被占用，启动器选择可用端口并将实际地址传给浏览器。多次启动默认聚焦现有实例；需要同时打开多个独立世界库时使用显式参数。

## 5. 程序、内容和状态目录

| 类别 | 内容 | 生命周期 | 可否导出 |
| --- | --- | --- | --- |
| 应用目录 | 可执行文件、前端、内置 schema、模板和虚构示例标准副本 | 随程序版本替换 | 否 |
| 世界库 | Markdown、YAML、地图和附件 | 用户长期拥有 | 是，核心内容 |
| 配置目录 | 最近世界库、界面偏好 | 随用户环境 | 可选，不进世界包 |
| 运行目录 | 索引、缓存、日志、锁文件 | 可删除重建 | 否 |
| 密钥存储 | 可选 AI/API 凭据 | 本机私密 | 永不导出 |

应用通过平台目录 API 获取默认位置，不在代码中硬编码某个用户名。用户可将世界库选择在普通文档目录、加密磁盘或可移动设备。网络共享目录和多设备同时写入不在首发支持范围内。

## 6. 导出功能

### 6.1 世界传输包

用于在另一台设备继续编辑。导出向导依次选择范围、内容状态、附件与目标文件，随后：

1. 扫描引用与依赖，展示缺失项和将被包含的内容。
2. 在暂存目录生成规范化清单。
3. 对每个内容文件计算 SHA-256。
4. 生成标准 ZIP 容器并使用 `.worldvault` 扩展名。
5. 重新打开成品，验证路径、文件数量、展开大小和校验和。
6. 报告保存路径、大小、范围以及是否包含草稿/废弃内容。

导出写入临时文件，验证通过后原子改名。目标磁盘空间不足或被中断时，不留下看似成功的半成品。

### 6.2 静态审阅包

静态审阅包是目录式 HTML ZIP，包含条目页面、导航、搜索所需静态索引和附件副本。它不启动后端、不包含编辑功能，不携带作者密钥和运行日志。页面显著标注导出时间、世界、分支、包含的状态以及“只读快照”。

审阅导出可以隐藏草稿、传闻、废弃内容或作者私密备注。首版不把 PDF 作为主要审阅格式，因为复杂链接、搜索和长篇条目更适合静态 HTML。

## 7. 导入流程

导入采用“检查—计划—暂存—提交”流程：

1. **检查**：识别格式版本，限制文件数量、单文件大小、总展开大小和压缩比；拒绝目录穿越、绝对路径、符号链接与校验失败。
2. **计划**：解析条目和依赖，与目标库按稳定 ID/哈希比较，生成新增、相同、冲突、缺失依赖列表。
3. **确认**：用户选择创建新世界库、合并，或取消；合并冲突必须明确处理。
4. **暂存**：在目标磁盘的临时目录完成 schema 迁移和全量校验。
5. **提交**：目标库加写锁，先建立恢复快照，再原子写入或替换受影响文件。
6. **验证**：重建索引并比较条目、关系、附件数量和哈希，最后生成导入报告。

任何一步失败都保持原世界库不变，并保留可读错误报告。暂存内容默认清理；为了诊断而保留时必须告知位置和隐私影响。

## 8. 版本兼容

应用版本、世界库 schema 版本和 `.worldvault` 格式版本分别管理。规则如下：

- 新应用读取受支持的旧格式，并在暂存区迁移副本；
- 原文件只有在导入确认后才进入新格式；
- 高于当前支持版本的包只展示元信息并拒绝导入；
- 导出界面显示最低兼容应用版本；
- schema 迁移必须有固定夹具和往返/回滚测试。

## 9. 构建与发布验证

发布门禁按顺序执行：

1. 单元测试、解析与导入安全测试。
2. 构建 wheel/sdist 并检查文件内容。
3. 在新虚拟环境、源码目录之外安装 wheel，测试统一 CLI 和真实健康接口。
4. 在目标操作系统构建独立目录可执行程序，测试 `--version`。
5. 断网运行发行程序，创建条目、保存、搜索、关闭并重新打开。
6. 设备/环境 A 导出 `.worldvault`，全新环境 B 导入，重建索引后比较内容哈希。
7. 扫描发行包，确认没有私人世界库、密钥、缓存和构建机路径。
8. 生成发行档案、SHA-256 校验文件和能力说明。

只有实际发行产物通过测试才算成功；从源码启动成功不能替代产物验证。

## 10. 故障恢复

- 程序无法启动：世界库仍是可读 Markdown，可换版本或普通编辑器打开。
- 索引损坏：删除运行索引并执行重建，不从传输包反向覆盖正文。
- 导入失败：撤销暂存，不改目标库；若已进入提交阶段，使用自动恢复快照回滚。
- 导出损坏：校验失败的文件不得标为成功；重新导出，不用它替代现有世界库。
- 程序升级失败：恢复旧应用目录，继续使用同一外部世界库；若 schema 已迁移，使用升级前快照。
