# ADR 0001：采用本地优先的 Markdown 真源

- 状态：已提议
- 日期：2026-07-16

## 背景

个人世界观可能维护多年，应用框架和数据库会变化甚至停止维护。设定正文必须保持可读、可迁移、可比较，并能被 Git 和普通备份工具处理。同时，Wiki 需要全文搜索、反向链接、关系和一致性检查，单靠逐文件扫描无法提供稳定体验。

## 决策

正文、元数据、关系声明和资源引用保存在 Markdown/YAML 文件中。SQLite 只保存从这些文件解析出的索引、缓存和检查结果。系统必须提供完整重建命令，并把“删除数据库后能恢复全部派生数据”作为发布测试。

应用默认本机运行并仅监听回环地址。外部服务和 AI 不是读取或编辑正文的必要依赖。

## 结果

优点：

- 内容不被应用锁定，可用任意编辑器和 Git 管理；
- 数据库损坏不会导致设定正文丢失；
- 能在不启动应用时检查、备份和迁移内容；
- 后续可以更换 UI 或索引实现。

代价：

- 必须解决文件与索引的同步、外部编辑和写入冲突；
- 文件格式变更需要兼容迁移；
- 复杂关系的编辑体验需要在可读文件与结构化数据之间权衡；
- 搜索和关系查询最终一致，而非所有写入都依赖单个数据库事务。

## 被否决的方案

- **仅数据库存储**：查询方便，但正文可移植性和长期可读性不足。
- **纯 Markdown、无索引**：实现简单，但中文检索、反向链接和一致性检查会随规模增长而变慢。
- **远程 SaaS 优先**：引入账户、网络、托管和隐私负担，不符合个人本地工具的主要场景。
