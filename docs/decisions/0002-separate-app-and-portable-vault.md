# ADR 0002：分离应用发行包与便携世界库

- 状态：已提议
- 日期：2026-07-16

## 背景

系统需要在其他设备下载即用，同时允许用户便捷传递世界观数据。若程序、索引和私人正文混在同一目录，升级可能覆盖数据，发行时也可能意外带出私人内容。单纯复制文件夹虽然开放，但无法可靠说明格式版本、文件完整性和合并冲突。

## 决策

应用发布为不含私人世界观数据的目标平台原生发行包。首发普通用户产物采用独立目录压缩包，内含运行时和全部必要静态资源，不要求预装开发环境。

用户世界库位于应用目录之外。跨设备编辑迁移使用 `.worldvault`：它是带版本清单和逐文件 SHA-256 的标准 ZIP 容器，不含索引、缓存、日志、密钥和 Git 元数据。导入必须先暂存、预览冲突、验证，再提交。

另提供静态 HTML 审阅包，用于无需安装应用的只读浏览；它不能作为可无损导回的编辑格式。

## 结果

优点：

- 程序可以升级或删除而不影响用户内容；
- 正式发行包能够自动证明不含测试者私人数据；
- 跨设备传输可以验证完整性和格式兼容性；
- 合并冲突基于稳定 ID 和内容哈希，而非脆弱的文件名；
- 不安装应用也能浏览静态审阅包。

代价：

- 需要维护格式版本、导入安全限制、暂存和回滚流程；
- 每个操作系统必须分别构建和验证原生发行物；
- 目录式发行包的文件多于单文件程序；
- 静态审阅包与可编辑传输包是两个不同产物，需要清晰解释。

## 被否决的方案

- **发行包内自带可写 `vault/`**：容易在升级、解压和分享程序时混入或覆盖私人数据。
- **只支持复制目录**：缺少版本清单、完整性检查和冲突预览，不够适合作为产品级迁移流程。
- **只提供单文件冻结程序**：在受限临时目录、资源定位和启动性能方面风险更高，首发可靠性不如目录式发行。
- **把 SQLite 数据库作为传输格式**：包含可重建状态，跨版本兼容和局部合并困难，也削弱 Markdown 真源原则。

## 重新评估条件

若后续构建工具能在全部目标平台稳定生成并验证单文件产物，且资源定位、启动速度和受限临时目录测试均通过，可以新增单文件下载项，但不得取代程序与用户世界库分离的边界。
