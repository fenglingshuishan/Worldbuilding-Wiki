# 产品设计

## 1. 产品定义

Worldbuilding Wiki 是一个仅服务于个人创作者的架空世界观工作台。它不是普通笔记软件的换皮，也不把所有知识强行画成一张关系图；它要解决的是长期世界构建中的四类问题：设定分散、信息难找、时间与关系冲突、作者真相和世界内说法混在一起。

产品默认中文、离线、本机运行。内容采用 Markdown 保存，即使应用停止维护，也能用任意文本编辑器继续使用。程序本体不携带具体世界观数据，用户可在任意受支持设备上首次运行时新建、打开或导入自己的世界库。

## 2. 设计原则

1. **内容归作者所有**：正文文件不使用专有二进制格式。
2. **正史必须可辨认**：已确认事实、候选设定、传闻和废弃设定不能混为一谈。
3. **先写作，后结构化**：最低要求只有标题和类型；其他字段可以逐步补充。
4. **关系具有时间性**：人物的归属、职位、同盟和敌对关系都可能随时间改变。
5. **索引可以重建**：数据库损坏不能导致正文丢失。
6. **AI 只能提案**：AI 生成内容进入草稿或变更建议，不能静默覆盖正史。
7. **程序与内容分离**：升级、替换或删除程序不应影响用户世界库。
8. **迁移先验证再写入**：导入必须预检格式、校验和、版本与冲突。

## 3. 核心使用场景

### 3.1 捕捉设定

创作者用快捷入口新建人物、地点、事件或自由笔记。应用按模板生成条目，自动建立唯一 ID；作者可以先只写一句话，稍后再补完整。

### 3.2 探索关联

正文使用 `[[页面标题]]` 建立自然链接。阅读一个人物时，侧栏展示反向链接、直接关系、参加的事件、所在地点以及待解决冲突。

### 3.3 编排历史

事件既能使用明确日期，也能使用“某纪元约 120—125 年”之类的不确定范围。时间线按统一内部序号排序，再转换为世界内历法显示。

### 3.4 管理不同真相

同一个事件可以同时存在：

- 作者确认的事实；
- 某个角色或组织的公开说法；
- 可靠性未知的传闻；
- 尚未采用的创作方案；
- 平行世界或分支时间线中的版本。

读者视角与作者视角分离。个人版本先实现作者视角，但数据模型不把“叙述”误当作“事实”。

### 3.5 检查一致性

系统报告断链、孤立条目、重复别名、时间倒置、同一时刻出现在相距过远地点、引用废弃设定等问题。报告必须给出证据链接，由作者决定是否修改。

### 3.6 跨设备传递

创作者在设备 A 将一个世界或整个世界库导出为 `.worldvault`，复制到设备 B 后直接用应用打开。应用先显示包的来源、版本、世界数量、文件数量、大小与冲突，再让用户选择“创建新世界库”或“合并到现有世界库”。

程序发行包和 `.worldvault` 是两类文件：前者包含应用、内置模板、文档和明确标记的虚构示例标准副本，后者包含用户世界库的当前内容。发布程序时必须自动检查发行包中不存在 `vault/` 私人正文、运行日志、密钥或历史导出。

### 3.7 导出浏览与审阅

导出入口提供两种明确用途：

- **世界传输包**：完整保留可继续编辑的数据，用于换设备、交接给自己的另一套环境或安全归档当前快照。
- **静态审阅包**：生成无需安装应用即可浏览的只读 HTML，适合在另一台设备审阅或临时分享；不承诺可无损导回编辑库。

导出时可选择整个世界库、单个世界或指定条目及其依赖。系统展示将包含的草稿、传闻、废弃设定和附件，避免把未准备好的内容意外带出。

## 4. 信息架构

一级入口保持稳定：

| 入口 | 用途 |
| --- | --- |
| 首页 | 最近编辑、待完善条目、未处理冲突和快捷新建 |
| 条目 | 按类型、标签、状态和世界筛选全部内容 |
| 时间线 | 查看事件、人物生命期和关系有效期 |
| 地图 | 浏览地图、地点层级和标记；MVP 后实现 |
| 关系 | 以列表为主、图谱为辅地探索实体关联 |
| 检查 | 集中处理断链、冲突和数据质量问题 |
| 设置 | 世界、历法、模板、存储和备份设置 |
| 迁移 | 新建/打开世界库，导入传输包，导出传输包或静态审阅包 |

编辑页采用三栏布局：左侧是导航与搜索，中间是 Markdown 编辑/阅读区，右侧是元数据、反向链接、关系和检查结果。窄屏时右栏折叠为抽屉。

## 5. 条目类型

MVP 提供以下基础类型，但允许用户扩展模板：

| 类别 | 示例 |
| --- | --- |
| 人物 | 君主、旅人、神祇、历史人物 |
| 地点 | 世界、大陆、国家、城市、建筑、自然地貌 |
| 组织 | 国家、家族、教团、公司、军队、秘密结社 |
| 事件 | 战争、迁徙、发明、灾难、条约、个人经历 |
| 群体 | 物种、民族、社会阶层、职业群体 |
| 文化 | 宗教、语言、习俗、制度、艺术 |
| 规则 | 魔法、科技、经济、法律、自然规律 |
| 物件 | 神器、书籍、交通工具、资源、货币 |
| 概念 | 术语、主题、未归类但稳定的设定 |
| 来源 | 作者笔记、灵感来源、世界内文献 |

类型是检索和模板工具，不是不可跨越的分类墙。一个条目只能有一个主类型，但可通过标签和关系表达多重身份。

## 6. 正史与叙述层级

每个条目有一个内容状态：

| 状态 | 含义 | 默认参与一致性检查 |
| --- | --- | --- |
| `canon` | 作者确认的当前正史 | 是 |
| `draft` | 尚未确认的候选设定 | 否，单独提示 |
| `rumor` | 世界内流传的说法 | 否 |
| `deprecated` | 已废弃但为追溯保留 | 否，检查正史引用 |

“传闻”不等于低质量正文。它应记录说话者、来源和可信程度，而不是被系统自动判真伪。平行时间线使用独立的 `branch` 字段，不通过复制整个知识库实现。

## 7. 搜索与发现

MVP 搜索范围包括标题、别名、正文、标签和结构化关系，支持中文全文检索。结果展示命中片段、类型、状态和所属世界。搜索框也作为命令入口，例如快速新建条目或跳转到时间线。

系统同时提供：

- 未建立页面的 Wiki 链接；
- 没有任何入链或出链的孤立条目；
- 同名和近似别名候选；
- 最近修改和长期未完成草稿。

## 8. AI 辅助边界

AI 是可选增强项，不属于第一阶段必需能力。允许的模式包括：

- 基于选定正史条目回答问题，并列出引用来源；
- 从正文建议元数据、关系或待补问题；
- 比较新草稿与现有设定，提出冲突候选；
- 按作者指定约束生成多个创意方案。

所有输出默认进入建议区。写入正文前显示差异并要求确认；密钥只存本机环境变量；未明确选择的私人内容不得上传到外部模型。

## 9. MVP 不做的事情

- 多用户权限、实时协作和公共社区；
- 公开网站发布与读者门户；
- 3D 世界、完整 GIS 或专业制图；
- 自动替作者决定正史；
- 为展示效果而加载无法阅读的大型全局关系图；
- 将图片、附件或正文锁进数据库。
- 让不同操作系统直接共用同一个正在打开的网络盘世界库；跨设备同步先采用显式导出、导入。
- 将传输包宣传成包含历史版本的完整备份；默认传输包只保存导出时的内容快照。

这些能力只有在个人写作流程证明需要后才进入路线图。

## 10. 成功标准

第一版成功不以功能数量衡量，而以以下任务是否顺畅衡量：

1. 30 秒内创建一个新人物并链接其出生地。
2. 从任何条目两步内看到反向链接和直接关系。
3. 删除索引数据库后，能从 Markdown 完整重建搜索和关系。
4. 能表达模糊日期、多个历法和有起止时间的关系。
5. 系统能指出冲突，但未经确认不会改变任何正史内容。
6. 整个设定库能通过复制目录或 Git 克隆迁移到另一台机器。
7. 新设备无需安装 Python、Node.js 或数据库服务即可启动发行版。
8. 在设备 A 导出的世界传输包可在干净的设备 B 导入，条目、关系、附件和校验和保持一致。
9. 导入发生冲突时先给出预览，不会覆盖目标世界库中的现有版本。
