# 实施路线

## 当前实施状态

截至 2026-07-16，M0 和 M1 已实现并由自动化测试、wheel 新环境运行和 Linux x64 独立程序验证。M2 已实现关系、时间范围、分支标注、声明存储、事件顺序/地点冲突检查和局部图 API；分支继承比较与可视化关系画布仍是增强功能。M3 已实现附件、静态 HTML 审阅包、导入导出和跨平台构建流水线；可交互地图标记、字段模板界面和可选 AI 不属于当前完成版本的必需运行路径。

## 1. 开发前需要确认的产品选择

设计默认值已经给出；只有进入编码前才需要最终确认：

1. **内容仓库**：开发期示例 `vault/` 可放在仓库中，但正式发行后用户世界库始终位于外部用户数据目录；应用仓库只保留匿名测试夹具。
2. **使用入口**：默认由原生启动器打开本机浏览器 Web 应用；不先做完整桌面壳或手机应用。
3. **AI 能力**：默认第一版关闭；基础 Wiki 与一致性检查稳定后再接入可替换模型适配器。

## 2. 里程碑

### M0：可行性原型

目标是验证最容易返工的环节，不追求可持续使用。

- 读写带 YAML 元数据的 Markdown；
- 解析 `[[ID|显示名]]` 与 `[[标题]]`；
- 从 100—1000 个中文条目重建索引；
- 验证中文标题、别名和正文检索；
- 验证外部编辑后的增量刷新。
- 验证运行时资源可从源码、安装包和冻结目录正确定位；
- 验证最小 `.worldvault` 导出后能在空目录无损导入。

退出条件：正文可无损往返，删除 SQLite 后可完整重建，搜索体验可接受。

### M1：可日常使用的个人 Wiki

- 首页、条目列表、阅读、编辑、新建和删除保护；
- 人物、地点、组织、事件、规则和概念模板；
- Wiki 链接、反向链接、标签、别名和全文检索；
- 草稿、正史、传闻和废弃状态；
- 原子保存、冲突检测、手动备份与索引重建；
- 断链、孤立条目、别名歧义检查。
- 完整世界库和单个世界的 `.worldvault` 导入、导出；
- 导入预检、校验和验证、创建新世界库和安全失败回滚；
- Windows x64 独立目录发行包，不要求预装开发运行库。

退出条件：可以仅靠应用完成一周真实设定整理，能用普通编辑器继续维护同一批文件，并能把发行包和世界传输包分别复制到干净 Windows 设备完成启动与迁移。

### M2：世界构建能力

- 一等关系编辑器和关系历史；
- 多历法、模糊日期、事件范围和时间线；
- 声明、来源与可靠性；
- 分支时间线和差异比较；
- 时间、地点、关系和废弃引用的一致性规则；
- 关系局部图，默认只展开一到两层。

退出条件：能完整表达一个人物跨越多个组织、地点与时代的经历，并解释每条冲突报告的证据。

### M3：创作增强

- 地图图片、图层和地点标记；
- 批量导入、静态只读导出；
- 模板和字段自定义界面；
- 可选 AI 检索问答、元数据建议和冲突解释；
- 独立桌面启动方式与跨平台打包评估。
- 静态 HTML 审阅包、选择性导出和合并冲突界面；
- Linux x64 独立发行包；macOS 发行包在具备原生构建、签名和验证环境后提供。

## 3. M1 验收场景

| 编号 | 场景 | 验收结果 |
| --- | --- | --- |
| A1 | 新建人物“林烬”，填写别名并链接雾港 | 文件格式合法，阅读页可跳转，雾港显示反向链接 |
| A2 | 将人物标题改为“林烬·阿措” | ID 链接保持有效，旧标题可选作别名 |
| A3 | 输入尚不存在的 `[[灰塔]]` | 显示未创建链接，可一键按模板建页 |
| A4 | 两个条目使用同一别名 | 检查页报告歧义，原文不被自动修改 |
| A5 | 用外部编辑器修改文件 | 应用检测变化并刷新索引，不覆盖新内容 |
| A6 | 删除 `runtime/index.sqlite` | 重启或执行命令后完整恢复搜索与反向链接 |
| A7 | 渲染含脚本的 Markdown | 危险脚本不会执行 |
| A8 | 复制 `vault/` 到新目录 | 配置新路径后无需数据转换即可打开 |
| A9 | 在设备 A 导出 `.worldvault` | 包中包含格式清单和全部内容校验和，不含索引、日志、密钥或绝对路径 |
| A10 | 在空白设备 B 导入传输包 | 条目、关系、附件和分支完整，重建索引后数量与来源一致 |
| A11 | 导入含相同 ID、不同正文的包 | 先展示冲突，不覆盖目标正文，可选择保留或另存草稿 |
| A12 | 在无 Python/Node.js 的 Windows 环境运行发行包 | 解压后启动成功，可创建、保存、关闭并重新打开世界库 |
| A13 | 发行包扫描 | 不包含真实 `vault/`、个人导出、缓存、日志、密钥和测试者绝对路径 |

## 4. 测试策略

- 解析器使用固定样例和往返测试，确保 Markdown 不被意外改写。
- Domain 层覆盖关系类型、模糊时间和分支继承的单元测试。
- Vault 层测试原子写入、并发版本冲突、非法路径和中断恢复。
- Indexer 测试全量重建与增量结果一致。
- 关键用户流程做浏览器端端到端测试。
- 用一份匿名示例世界作为验收数据，不直接拿私人设定做公开测试夹具。
- 构建产物离开源码目录后测试，防止误用开发目录中的静态资源。
- 每个操作系统原生构建并测试对应可执行文件；不能把 Linux 产物作为 Windows/macOS 产物交付。
- 跨设备回环测试执行“创建内容 → 导出 → 全新环境导入 → 重建索引 → 比对哈希”。

## 5. 第一轮开发任务建议

1. 建立最小 Python 项目与命令行入口。
2. 定义条目 schema、错误模型和测试样例。
3. 完成只读扫描、解析和索引重建。
4. 完成搜索、条目阅读和反向链接页面。
5. 完成安全写入与编辑冲突检测。
6. 加入模板、新建流程和最小检查器。
7. 用验收场景 A1—A8 做发布门禁。
8. 实现传输包、导入暂存和 A9—A11 迁移门禁。
9. 建立 Windows 原生构建流水线和 A12—A13 发行门禁。

在 M0 原型通过前，不投入地图、AI、桌面打包或复杂图谱。
